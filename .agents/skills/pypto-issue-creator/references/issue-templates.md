# Issue 模板

本文档定义了 5 种 Issue 类型的标准模板，对齐 GitCode 平台 `cann/pypto` 仓库的 ISSUE 处理规约（#350）。

---

1. [Bug Report](#1-bug-report)
2. [Feature Request](#2-feature-request)
3. [Documentation](#3-documentation)

5. [Task](#5-task)

---

## 1. Bug Report

**触发场景**: 报错、异常、失败、精度问题

**必填字段**: 标题、问题描述、环境信息、重现步骤、预期结果、日志/截图

**内容规范**：
- 根据问题类型，在对应字段中提供有针对性的信息
- 常见场景示例：
  - **编译 / 构建失败**：在日志中提供完整编译输出，在问题描述中标注编译命令和目标平台
  - **运行时错误 / AICore Error**：在重现步骤中附上完整可运行的复现脚本（最小化示例优先）；在日志中提供完整错误堆栈，关键行可用注释标注
  - **精度问题**：在重现步骤中附上复现脚本和输入数据规格；在日志中提供实际输出与期望输出的对比
  - **安装 / 环境问题**：确保环境信息填写完整，在日志中提供安装命令和完整报错

```markdown
### Describe the current behavior / 问题描述 (Mandatory / 必填)

[清晰简洁地描述问题]

### Environment / 环境信息 (Mandatory / 必填)

- **服务器/NPU 型号**: [Ascend 910B / Ascend 910C / A3 / A2]
- **PyPTO 版本/Commit**: [commit hash]
- **CANN 版本**: [版本号]
- **Python 版本**: [版本号]
- **操作系统**: [版本号]

### Steps to reproduce the issue / 重现步骤 (Mandatory / 必填)

1. [步骤1]
2. [步骤2]
3. [步骤3]

### Describe the expected behavior / 预期结果 (Mandatory / 必填)

[描述应该发生什么]

### Related log / screenshot / 日志 / 截图 (Mandatory / 必填)

[粘贴相关日志或报错信息]

### Special notes for this issue/备注 (Optional / 选填)

[任何其他有助于理解问题的信息]
```

## 2. Feature Request

**触发场景**: 希望支持、新增功能、优化、扩展

**必填字段**: 标题、背景信息、信息来源

**标题前缀**: `[Requirement|需求建议]:`

**内容规范**：
- 根据需求类型，在对应字段中提供可量化、可验证的补充信息
- 常见场景示例：
  - **新功能 / API 支持**：在 Background 中参考 PyTorch、Triton、cuTile、TileLang 等框架的同类实现；在 Design 中提供期望的 API 示例或伪代码
  - **性能优化**：在 Background 中提供基准性能数据，在 Benefit 中给出目标性能指标
  - **架构 / 流程改进**：在 Background 中提供当前架构说明，在 Design 中给出改进方案
  - **兼容性 / 版本适配**：在 Background 中提供目标版本和平台信息，在 Benefit 中说明兼容性需求

```markdown
### Background（背景信息） (Mandatory / 必填)

[描述新需求的功能是什么，解决了什么问题。可补充：为什么需要这个功能？现有功能为什么不够？]

### Origin（信息来源） (Mandatory / 必填)

[说明是哪个部门/团队/场景提出的需求，以便确定优先级]

### Benefit / Necessity（价值/作用） (Optional / 选填)

[描述该需求的价值和应用场景。对现有功能有何提升？]

### Design（设计方案） (Optional / 选填)

[描述设计的总体思路，可使用伪代码]
```

## 3. Documentation

**触发场景**: 文档缺失、文档错误、文档改进

**必填字段**: 标题、文档链接、问题文档片段

**标题前缀**: `[Documentation|文档反馈]:`

**内容规范**：
- 根据问题类型，在对应字段中提供有针对性的信息
- 常见场景示例：
  - **文档缺失**：在 Issues Section 中说明期望的文档主题和应覆盖的内容；在 Document Link 中给出最相关的文档位置
  - **描述错误 / 过时**：在 Issues Section 中粘贴有问题的文档片段，在 Suggested Fix 中提供修正建议或修正后的内容
  - **示例代码问题**：在 Issues Section 中粘贴不可运行的示例，在 Suggested Fix 中提供可运行的修正版本

```markdown
### Document Link（文档链接）

[需要改进的文档链接]

### Issues Section（问题文档片段）

[粘贴有问题的文档片段]

### Existing Issues（存在的问题）

[描述文档存在的问题]

### Suggested Fix / 修正建议 (Optional / 选填)

[描述应如何修改，或直接提供修改后的内容]
```

## 4. Question

**触发场景**: 如何使用、为什么、不理解

**必填字段**: 标题、问题描述

**内容规范**：
- 根据问题类型，在对应字段中提供有针对性的上下文
- 常见场景示例：
  - **使用方式 / 配置方法**：在问题描述中说明目标效果和当前操作，附上相关代码片段或配置；如已尝试排查，在"已尝试的方法"中说明
  - **功能 / API 咨询**：在问题描述中说明期望实现的功能和业务场景

```markdown
### 问题描述

[清晰描述你的问题]

### What have you tried / 已尝试的方法 (Optional / 选填)

[描述已经尝试过的排查步骤或方法，避免重复建议]
```

## 5. Task

**触发场景**: 需要开发、计划、任务

**必填字段**: 标题、任务描述、任务目标

**内容规范**：
- 根据任务类型，在对应字段中提供有针对性的上下文
- 常见场景示例：
  - **新模块 / 功能开发**：在任务描述中说明功能需求、设计思路和涉及的技术模块
  - **版本 / 平台适配**：在任务描述中说明目标版本、平台信息和兼容性要求
  - **测试补充**：在任务目标中明确测试覆盖率要求、需覆盖的场景
  - **环境 / 工具搭建**：在任务描述中说明环境需求和预期产出

```markdown
### 任务描述

[清晰描述需要完成的任务]

### 任务目标

1. [目标1]
2. [目标2]
3. [目标3]
```
